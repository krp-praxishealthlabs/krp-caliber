## Claude CLI smoke test

Command: `claude -p "respond with the single word ok"` (run via the Bash tool inside this Claude Code session).

```
ok
SessionEnd hook [caliber refresh --quiet] failed: Hook cancelled
SessionEnd hook [caliber learn finalize --auto] failed: Hook cancelled
claude -p "respond with the single word ok" 2>&1  3.91s user 2.32s system 38% cpu 16.204 total
```

**Observations:**
1. The actual response (`ok`) returned successfully — the model call works.
2. But the run took **16 seconds wall-clock** — most of that is hook timeout.
3. **Two SessionEnd hooks were cancelled by Claude Code mid-execution**: `caliber refresh --quiet` and `caliber learn finalize --auto`. Both visible in stderr.

## Why this matters for the audit

`.claude/settings.json` registers two SessionEnd hooks that each invoke caliber commands which themselves spawn `claude -p` for refresh / finalize work. When I run `claude -p` interactively (NOT spawned by caliber), Claude Code fires those SessionEnd hooks with no `CALIBER_SUBPROCESS=1` env marker. The hooks try to do real work (refresh requires a model call, finalize requires an LLM call), Claude Code's hook timeout fires first, the user sees "Hook cancelled" errors.

**The subprocess sentinel only protects against the SECOND level of recursion** (caliber-spawned `claude -p` → SessionEnd → caliber → would spawn `claude -p` again, but the second-level caliber sees `CALIBER_SUBPROCESS=1` and short-circuits). It does **NOT** protect against the FIRST level (user's `claude -p` → SessionEnd → caliber tries to do real work → timeout).

This is reproducible 100% of the time on this machine. Every interactive `claude -p` session in this repo emits "Hook cancelled" twice in stderr. For a user, this is daily-friction-level noise that erodes trust. They'd reasonably wonder "is caliber broken?" — even though the actual model response was fine.

This belongs in the findings doc as a **P0 user-experience finding** distinct from the audit's "install reliability" findings. It's not a bug in install — it's a side-effect of *being installed*.

## Decisions for the live tests

- **Use `node /Users/alonpe/personal/caliber/dist/bin.js` for all caliber invocations** during the audit. This tests source 1.48.2 behavior directly.
- For paths that explicitly model the "fresh user" experience (Phase 2 Path A on test repos), accept that bootstrap and init will *install* hooks pointing to whatever `resolveCaliber()` decides. Capture which path it picks per-repo and surface as an observation.
- Do NOT run `caliber config` to override the installed-via-PATH version, because we want the hook scripts to reflect what a real user upgrading from 1.40.4 → 1.48.2 would experience.
- For the npx-fresh path (which the README leads with), use `npx @rely-ai/caliber@1.48.2 init` once during Phase 2 to capture the published-package experience.
- **Pre-flight passes.** All preconditions are met; smoke test confirmed model auth works despite hook cascade noise.

## Pre-existing P0 findings from pre-flight (before any live test runs)

- **[P0] Every interactive `claude -p` session in a caliber-equipped repo emits "Hook cancelled" twice in stderr.** Cancellation is from Claude Code timing out the SessionEnd hooks. Visible to the user. Adds 10–15 seconds of latency per `claude -p` invocation. Reproduced 100% on this machine. The subprocess sentinel only catches recursion from caliber-spawned claude sessions, not user-initiated ones.
  - Suggested fix: SessionEnd hooks should detect "no claude-spawned context" via something OTHER than CALIBER_SUBPROCESS — perhaps check whether the parent `claude` process is the user's interactive session vs a caliber-spawned one. Or: SessionEnd hooks should be `--no-llm` mode by default and only do work that doesn't require model calls (e.g. update state file timestamps); deferred LLM work to next caliber invocation.
  - This finding alone makes the messaging redesign moot — fix this BEFORE polishing anything.
